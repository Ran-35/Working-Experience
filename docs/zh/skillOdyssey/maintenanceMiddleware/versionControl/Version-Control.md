# VSC 版本控制系统
<br>

## 什么是版本控制系统？
版本控制系统通过记录文件或代码的变化，并允许用户查看、恢复、合并不同版本，来管理文件和目录的版本历史。
* **历史记录**：记录每次文件修改的详细信息，包括修改的内容、修改人、修改时间等。
* **协作支持**：多个开发者可以同时修改不同部分的文件，系统会自动记录和合并这些修改。
* **备份与恢复**：可以随时回退到某个历史版本，避免意外丢失或错误修改。
* **并行开发**：通过分支和合并，开发者可以独立地进行开发，最终将更改合并到主代码库中。
<br>

## 版本控制系统的分类？  
### 1. 集中式版本控制系统（CVCS）
所有的版本历史都存储在一个中央服务器上，开发者从这个服务器获取文件的副本，并在本地进行修改。当完成修改后，再将更新提交回中央仓库。  

**特点：**
* 中央仓库：所有文件的历史记录存储在一个中央服务器上，所有操作（提交、更新）都与中央仓库交互。
* 本地副本：开发者的工作副本只是中央仓库的快照，提交和更新必须依赖网络连接。
* 易于管理和备份：因为所有数据都集中在服务器上，管理员可以容易地对整个项目进行备份和管理。
* 适用于小型团队：适合那些开发较简单、团队规模较小的项目。  

**缺点：**
* 依赖网络：没有网络连接时，开发者不能完成提交、更新等操作。
* 分支操作不灵活：分支创建和合并相对繁琐，容易产生冲突。  

**常见集中式版本控制系统：**
* SVN（Subversion）
* CVS（Concurrent Versions System）
* Perforce（虽然支持分布式，但通常用作集中式 VCS）
<br>

### 2. 分布式版本控制系统（DVCS）
每个开发者的本地机器上都有一个完整的代码仓库，包括项目的所有历史版本。开发者可以离线工作，提交到本地仓库，等网络连接恢复时，再将更改推送到中央服务器。  

**特点：**
* 完全离线工作：每个开发者的机器上都有完整的版本历史，可以随时查看历史、提交更改和创建分支。
* 强大的分支和合并：分支和合并操作更加高效和灵活，支持并行开发。
* 分布式存储：每个开发者的机器都是一个完整的仓库，避免了对中央服务器的依赖。
* 适合大型团队：适合分布式开发，尤其是跨地域的团队协作。  

**缺点：**
* 本地仓库大：每个开发者都有完整的历史记录，工作副本可能非常大。
* 管理复杂：对于不熟悉分布式系统的用户，Git 这样的系统可能会有较陡峭的学习曲线。  

**常见集中式版本控制系统：**
* Git
* Mercurial
* Bazaar
